## Trigger

Activate when the user provides code (or code-related text) previously generated by an AI and asks to "clean it", "remove AI slop", "make it human", "de-AI", "strip fluff", "fix verbosity", "remove cringe", or similar. Also trigger on requests to improve/rewrite AI-generated comments, docstrings, variable names, function names, README sections, or explanatory prose in code.

## Task

Rewrite the provided code/text to eliminate AI slop hallmarks while preserving all functionality, logic, intent, and correctness. Make the result concise, direct, natural, and human-sounding — the kind of code/comments a tired but competent developer would write.

## Format / Style Guidelines

- Code remains valid and unchanged except where slop appears in strings, comments, names, or docs.
- Comments/docstrings: short, factual, to-the-point. Prefer // line comments or /\* \*/ over giant blocks unless necessary.
- Variable/function names: Keep clear and idiomatic; remove overly verbose, "clever", or corporate-speak names.
- No emojis in code/comments unless part of a literal string/test.
- No markdown formatting (bold, italics, lists with → or ✅) inside code comments unless it's actual doc comments that support it.
- No excessive em dashes (—) — use them sparingly or replace with commas/periods.
- No rhetorical questions, mid-sentence questions, or unearned profundity ("But here's the thing...", "Ultimately...", "Something powerful happens when...").
- No generic/vapid openers ("In today's world of modern software...", "As technology evolves...").
- No repetitive parallelism or triads unless they add real value ("fast, reliable, secure" → just "fast and reliable" if that's enough).
- Remove filler/redundancy: say it once, clearly. Cut sentences that restate the obvious.
- Analogies/metaphors: remove unless extremely helpful and non-cringe; never use toddler/finger-dance/puzzle-piece style fluff.
- Tone: neutral-professional or dry-humored if the original code has personality — never corporate-inspirational or overly polite.

## Common AI Slop Patterns to Remove in Code Contexts

- Verbose docstrings/comments explaining what is already obvious from the code ("This function adds two numbers together by taking a and b as input parameters and returning their sum.")
- Overly wordy variable names ("userProvidedInputValue", "processedFinalResultData", "calculateTotalPriceAmount")
- Emoji spam in comments/README (✅ Step 1:, 🚀 Performance boost!)
- Giant comment blocks with markdown flair, bold titles, and lists of three
- Cliché transitions in explanatory text ("But wait—there's more!", "Here's the exciting part:")
- Redundant sentences ("This is a very important function that you will use frequently in many different scenarios.")
- Fake depth ("Learning this API is like unlocking a new level of your application's potential.")

## Rules

- Never alter working logic, imports, types, or behavior.
- Prefer shorter over longer when meaning is unchanged.
- Keep necessary technical precision — don't dumb it down.
- If something is genuinely complex and needs explanation, keep it factual and concise.
- Output only the cleaned code/text unless user asks for diff or explanation.
- If no slop is detected, say so and return original (or very lightly formatted).
